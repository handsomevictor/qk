# qk 全面审计与路线图报告

基于对 `qk` 项目的重新从零开始的深入代码审计与分析，以下是针对当前架构、性能、功能及代码质量的全面评估，以及未来的改进方向和新功能建议。

## 1. 架构与性能审计

**优势：**
* **单遍流式处理 (Streaming path):** 关键字层 (`fast_layer`) 在没有聚合或排序的情况下，可以通过 `stdin` 进行纯流式处理（`run_stdin_streaming_keyword`），内存复杂度为 O(1)，能够安全处理 2GB+ 的巨型日志。
* **零拷贝字符串抽象:** 使用 `Arc<str>` 作为字段名的 Interning（驻留），有效减少了重复字段名带来的内存开销。
* **错误容忍度:** 解析器在遇到损坏行（corrupt lines）时能优雅地发出警告并继续，而不是直接崩溃。

**核心架构缺陷与瓶颈 (🔴 需优先解决)：**

1. **全量物化 (Full Materialization) 导致的 OOM:**
   * **问题描述:** 只要用户直接传递文件路径（如 `qk where level=error app.log`）而不是使用管道（`cat app.log | qk`），或者使用了 **DSL 模式**，`main.rs` 就会调用 `load_records`，将整个文件的所有记录读入 `Vec<Record>` 驻留内存。
   * **影响:** 处理 1GB 的文件需要占用数 GB 的内存，在普通机器上极易触发 OOM (Out Of Memory)。
   * **改进建议:** 将统一的管道抽象为基于迭代器 (`Iterator`) 的流式处理引擎。让 DSL 和文件直接读取都能默认按行（惰性）求值，只有在遇到 `sort` 或 `group_by` 时才进行内存缓冲。

2. **阻塞式 `tail -f` (流式输入的挂起问题):**
   * **问题描述:** `qk` 的流式读取目前依赖于 `BufReader::lines`，虽然能逐行输出，但如果输入流（如 `tail -f`）没有发出 EOF，`qk` 内部在某些场景或退出机制上可能会无限阻塞。
   * **改进建议:** 彻底重构标准输入流的消费逻辑，使其成为一个完全非阻塞的实时流水线，并在内部实现早期退出信号（例如 `limit 5` 达到后立即退出进程）。

3. **`Value` 的深层拷贝与内存膨胀:**
   * **问题描述:** `Record` 底层使用 `serde_json::Value`。当处理海量数据时，相同的字符串值（例如 HTTP status `"200"` 或服务名 `"api"`）会在内存中被创建数百万次 `String`。
   * **改进建议 (延迟到下一个大版本):** 放弃 `serde_json::Value`，实现一个专用的零拷贝/字符串驻留（String Interning）的 `Value` 枚举，从根本上降低数据加载阶段的内存占用。

## 2. 功能性审计

**优势：**
* **双层查询语法:** 提供了无引号负担的快速关键字层，以及图灵不完备但足够强大的 DSL 层（支持嵌套字段、正则表达式、相对时间过滤、算术运算 `map`）。
* **多字段聚合:** 已经支持了 `count by level, service` 以及日历时间分桶（`count by day ts`）。

**缺失功能与扩展建议 (🟡 中优先级新功能)：**

1. **数组 (Array) 深度处理能力:**
   * **现状:** `qk` 目前对顶层结构化对象处理极佳，但如果日志中包含数组（如 `{"tags": ["prod", "web"]}`），除了基础的 `contains` 之外，缺乏深入操作。
   * **新功能需求:**
     * 在 DSL 中引入 `flatten` 或 `unnest` 操作，将数组展开为多条独立的记录（类似于 jq 的 `.[]`）。
     * 支持数组索引访问，如 `pick(.tags[0])`。
     * 支持对数组内部进行 `map` 或 `filter` 映射。

2. **跨数据流连接 (Joins / Lookups):**
   * **现状:** 如果提供多个文件，`qk` 只是将它们视为同一个数据流的追加（Append/Union）。不支持关系型 Join。
   * **新功能需求:** 引入类似 SQL 的 Lookup 功能（例如 `qk '| lookup(user_id, "users.csv", id) | ...'`），这在日志分析中通过关联静态字典表来富化数据是一个高频刚需。

3. **高级字符串提取与操作 (DSL 增强):**
   * **现状:** 目前 DSL 只有简单的 `replace`, `to_lower`, `to_upper`。
   * **新功能需求:** 
     * **正则捕获组提取:** 允许通过正则表达式从无结构的文本字段中提取新字段（如 `| map(.id = extract(.msg, "user_id=(\d+)"))`）。
     * **字符串拼接与切分:** 支持 `concat(.a, .b)` 和 `split(.path, "/")`。

4. **窗口函数与排序取 Top-N:**
   * **现状:** `group_by` 可以分组，但无法在分组内部进行排序并取前 N 个（例如“获取每个服务延迟最高的 3 条请求”）。
   * **新功能需求:** 在聚合阶段引入类似 SQL 的窗口机制或增强的 `top_n_by(.latency, 3)` 分组管道。

## 3. 测试与代码质量审计

**现状评估:**
* 代码库拥有 345 个测试用例，覆盖率极高，包括了 `tests/stability.rs` 中的 10k 记录稳定性测试和 `large_file.rs` 的 2GB 压力测试。代码质量高，通过了严格的 Clippy 检查。

**需要改进的地方 (🟢 低优先级维护):**
1. **自动化内存追踪测试:**
   目前的 `large_file` 测试因为耗时和资源问题被标记为 `#[ignore]`。建议在常规测试套件中引入自定义的 Global Allocator 包装器，针对几万条记录的小型数据集严格断言（Assert）内存分配峰值，以防止未来的 PR 无意中破坏 O(1) 的流式内存保证。
2. **错误输出的日志限流:**
   目前对 corrupt lines 的容错会每行打印一条 `[qk warning]`。如果一个千万行的文件有 10% 是坏行，错误流会刷屏。建议引入警告限流（Rate Limiting）机制，或者在批处理结束时只打印一句 "Skipped N corrupt lines"。

---

## 4. 行动总结

* **最大痛点 (P0):** 解决文件读取和 DSL 模式下的全量物化 (OOM) 问题，构建贯穿全局的 `Iterator` 流式处理管道。
* **最高价值新功能 (P1):** 在 DSL 中加入数组展平 (`unnest`) 和正则提取 (`extract`) 函数，以及跨文件的 `lookup/join` 能力，这将使 `qk` 的能力真正逼近 `jq` + `awk` 的完全体。
