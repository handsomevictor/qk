# qk 全面审计报告

基于对项目的深入审计（涵盖最新的 P0–P6 修复，以及最近的 DSL 边界测试、流式韧性测试、日历分桶功能和 DSL 时间属性提取等增强），以下是关于测试覆盖、性能与稳定性、功能性以及代码质量等方面的全面审计报告及可操作建议。

## 1. 测试覆盖情况

**优势与已覆盖领域:**
* **测试用例大幅增加:** 项目目前包含 315 个测试用例，全部通过。
* **DSL 边界测试得到完善:** 新增了针对深层嵌套、括号未闭合、缺少右值、空输入、长字段名、缺失字段、AND/OR 优先级等边缘情况的测试，显著提升了 DSL 层的防错能力。
* **时间属性与日历分桶测试:** 新增了 `hour_of_day`, `day_of_week`, `is_weekend` 的测试，以及基于日历单位（日、月、年、小时）的聚合测试 (`group_by_time` 与 `CalendarUnit`)，时间相关的边界覆盖（如 UTC 跨天）已相对完善。
* **流式韧性 (Streaming Resilience) 测试增强:** 针对 `stdin` 流式处理，增加了遇到损坏行 (corrupt lines)、全空输入、纯空行等场景的集成测试，验证了系统能跳过错误并打印警告而不会崩溃。

**剩余测试缺失与不足 (🟡 风险点):**
* **大文件与内存极限测试的自动化程度 (中风险):** 目前有 `tests/large_file.rs`（包含 2GB 流式测试和 200MB 批处理测试），但这些测试被标记为 `#[ignore]`，需要手动运行。在常规 CI 流程中，缺乏对 OOM 边界或大文件性能退化的自动保护网。
* **DSL 流式处理 (Streaming DSL) 测试缺失 (中风险):** 目前的流式韧性测试主要集中在 `fast_layer` (关键字层)。对于 DSL 模式下的流式处理（如果支持的话），或者 DSL 在处理极大单行 JSON 时的行为，仍缺乏明确的极限测试。

## 2. 性能与稳定性

**优势:**
* **内存分配优化:** `Record.raw` 改为 `Option<String>` 避免了聚合时的无用空字符串分配（P4）。
* **容错性提升:** 解析器（如 NDJSON）遇到损坏行时不再中断，而是记录警告并继续（P0 和近期的流式韧性修复）。

**剩余性能与稳定性风险:**

* **🔴 全量物化导致的 OOM 风险 (高风险):**
  在批处理模式下（指定文件参数或使用需要缓冲的查询如 `sort`、非时间分桶的 `group_by`），`load_records` 依然会将整个文件的所有记录解析为 `Vec<Record>` 并驻留内存。处理数 GB 的日志文件时，极易导致内存耗尽并崩溃。这依然是项目最大的架构瓶颈。
* **🟡 DSL 模式与文件输入的流式退化 (中风险):**
  当前的 O(1) 内存流式处理主要针对 `stdin` 且无缓冲的关键字查询（如 `qk where level=error`）。如果直接读取文件（如 `qk where level=error app.log`）或使用 DSL，系统可能会退化为全量读取模式，导致内存飙升。
* **🟢 [DEFERRED] 字符串内存膨胀 (低优先级 - 架构限制):**
  由于 `serde_json::Value::String` 内部使用 `String`，相同值的字符串在数百万条记录中会重复分配，导致内存膨胀。（**注: 对应 P6 问题，明确延后到下一个大版本，当前无需修改**）。

## 3. 功能性

**优势:**
* **时间操作大补全:** 实现了 `between` 运算符、相对时间 (`now-5m`)、日历单位分桶 (`count by day ts`)，以及 DSL 层的时间属性提取 (`hour_of_day`, `is_weekend` 等)。这极大增强了日志分析的表达能力。
* **错误提示优化:** DSL 解析错误现在会包含位置提示和上下文片段，显著改善了用户体验。

**剩余功能性风险与缺失:**

* **🟡 缺失部分聚合操作与数学函数 (中风险):**
  DSL 和关键字层目前支持 `sum`, `avg`, `min`, `max`，但缺少：
  * **去重计数**: `count_distinct`（如统计有多少个独立的 IP）。
  * **基本数学运算**: DSL 层缺少加减乘除运算符（如 `.latency / 1000` 将毫秒转为秒）。
* **🟢 逗号分隔符的语法歧义 (低风险):**
  关键字层将逗号视为 `and` 的别名。如果用户查询未加引号的包含逗号的字符串（例如 `where msg contains error,timeout`），可能会引发意外的解析截断。

## 4. 代码质量

**优势:**
* 零 Clippy 警告，代码格式规范，文档（包括中英文）与实现保持同步。
* `src/util/time.rs` 的抽象（`CalendarUnit` 等）和 DSL AST 节点的扩展非常清晰。

**潜在技术债:**
* 🔴 **解析与评估的强耦合 (高风险):** `load_records` 一次性返回 `Vec<Record>` 阻碍了全局流式架构的实现。未来需要将解析层彻底重构为基于 `Iterator<Item = Result<Record>>` 的管道模型。
* 🟡 **`Value` 类型锁定 (中风险):** 强依赖 `serde_json::Value` 限制了对内存布局（如字符串驻留、零拷贝引用）的底层优化能力（关联 P6）。

## 5. 剩余问题汇总与可操作建议

### 🔴 高优先级 (Immediate Action / 架构演进)

1. **统一的流式迭代器管道 (解决 OOM 风险)**
   * **影响:** 内存占用风险极高，限制了 `qk` 成为真正通用的大数据 CLI 工具。
   * **建议:** 重构引擎，使得 `qk where ... file.log` 和 `qk '...' < file.log` 都能默认使用惰性迭代器 (`Lazy Evaluation`) 处理，仅对 `sort` 和 `group_by` 进行必要的缓冲。

### 🟡 中优先级 (Short-term Improvements)

1. **实现去重计数 (Count Distinct)**
   * **影响:** 功能性缺失，用户无法回答“有多少个不同的用户访问过”此类常见问题。
   * **建议:** 在关键字层增加 `count_unique` 或在 DSL 层增加 `distinct` 操作符。
2. **DSL 支持基础算术运算**
   * **影响:** 无法在查询时进行单位转换（如 byte 到 MB）。
   * **建议:** 在 DSL 的表达式解析中引入 `+`, `-`, `*`, `/`。
3. **补充轻量级的自动化 OOM 防护测试**
   * **影响:** 稳定性回归风险。
   * **建议:** 在常规 `cargo test` 中加入使用内存分配器追踪的单元测试，验证单次管道执行后内存是否被正确释放，而无需每次都跑 2GB 的物理文件测试。

### 🟢 低优先级 (Long-term Bets & Deferred)

1. **[DEFERRED] 短字符串驻留 (String Interning)**
   * **影响:** 内存分配开销。
   * **状态:** 延后到下一个大版本。届时需替换 `serde_json::Value` 为自定义枚举。
2. **增强错误恢复日志的可视化**
   * **建议:** 当前跳过损坏行会打印 `[qk warning]`，在大文件中可能会刷屏。建议在处理结束时，输出一条总计警告（例如：“Skipped 150 corrupt lines. Use --verbose to see details”）。
