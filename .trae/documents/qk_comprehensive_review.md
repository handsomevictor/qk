# qk 全面审计报告

基于对项目的深入审计（涵盖最新的 P0–P5 修复），以下是关于测试覆盖、性能与稳定性、功能性以及代码质量等方面的全面审计报告及可操作建议。

## 1. 测试覆盖情况

**优势与已覆盖领域:**
* **核心与边界修复覆盖良好:** 近期的修复（P0-P5）伴随了大量测试。例如，`ndjson.rs` 增加了应对损坏行的恢复测试；`time.rs` 补充了 UTC 跨天、时区偏移、相对时间 (`now-5m`) 等 8 个边界测试；`eval.rs` 增加了 `Between` 逻辑和聚合静默失败修复的测试。目前总计有 280 个测试用例，覆盖了大部分功能逻辑。

**测试缺失与不足 (🔴 风险点):**
* **大文件与内存极限测试缺失:** 没有任何测试用例验证系统在处理超大文件（如 >2GB NDJSON）、超大单行 JSON 或包含巨大 JSON 数组时的行为。缺少 OOM (Out Of Memory) 边界测试和超长字符串字段解析的限制测试。
* **真实流式场景 (E2E Streaming) 测试不完善:** 尽管 `main.rs` 中有 `run_stdin_streaming_keyword` 以支持 `tail -f`，但现有测试（如 `streaming_filter_produces_correct_results`）仅测试了有限的静态输入，缺乏对“持续写入、阻塞等待新数据”这一真实流式场景的 E2E 验证，也没有验证管道在流式模式下遇到非法数据时的恢复能力。
* **DSL 层的健壮性与深度嵌套测试:** 虽然关键字层测试较多，但 DSL 层对于极端嵌套 JSON、超长表达式、缺少闭合括号等语法错误边缘情况的测试仍有欠缺。

## 2. 性能与稳定性

**优势:**
* 解决了 P4 (聚合记录的空字符串分配问题)，`Record.raw` 改为 `Option<String>`，有效减少了聚合操作时的内存开销。
* 解决了 P0 (损坏行导致解析中止)，现在会记录警告并继续，提升了稳定性。

**剩余性能与稳定性风险:**

* **🔴 全量物化导致的 OOM 风险 (高风险):**
  在批处理模式下（指定文件参数或使用需要缓冲的查询如 `sort`、DSL），`load_records` 依然会将整个文件的所有记录解析为 `Vec<Record>` 并驻留内存。处理数 GB 的日志文件时，这极易导致内存耗尽并崩溃。
* **🟡 字符串内存膨胀 (中风险 - 标记为 deferred):**
  由于 `serde_json::Value::String` 内部使用 `String`，即使相同的字符串值在数百万条记录中重复出现，也会发生重复分配。这会导致巨大的内存占用。（**注: P6 架构改动，延后到下一个大版本**）。
* **🟡 流式处理的局限性 (中风险):**
  目前的 O(1) 内存流式处理 (`run_stdin_streaming_keyword`) **仅限**于无缓冲的关键字查询且必须从 `stdin` 读取。一旦用户使用 DSL (`qk '.level=="error"'`) 或直接指定文件 (`qk where level=error app.log`)，程序就会退化为全量物化模式，导致流式处理失效且内存飙升。

## 3. 功能性

**优势:**
* 新增了 `between` 运算符和相对时间 (`now_secs()`)，极大地增强了时间过滤的实用性。
* 聚合函数 (`avg`, `min`, `max`, `sum`) 遇到空字段现在会正确返回 `Value::Null` 并警告，解决了之前的静默计算错误 (P1)。

**剩余功能性风险与缺失:**

* **🟡 高级日期时间函数缺失 (中风险):**
  虽然有了 `between` 和相对时间，但仍缺乏用于复杂报表和数据清洗的时间操作，例如：
  * **`date_trunc`**: 无法将时间戳截断到天、月、年（目前 `group_by_time` 只能按固定秒数/时长间隔分桶，无法自然按自然月/自然年聚合）。
  * **时间属性提取**: 缺少如 `is_weekend`, `day_of_week`, `hour_of_day` 等函数，导致无法进行“统计周末错误率”等常见分析。
* **🟢 DSL 错误提示不友好 (低风险):**
  DSL 解析器遇到复杂语法错误时，给出的提示往往不够精确，难以帮助用户快速定位是缺少括号还是拼写错误。

## 4. 代码质量

**优势:**
* 模块划分清晰（`detect`, `parser`, `query`, `output`）。
* `cargo clippy -- -D warnings` 零警告，代码风格良好。
* 良好的注释和 `--explain` 模式提示 (P5)。

**潜在技术债:**
* 🔴 **解析与评估耦合 (高风险):** 批处理模式下 `load_records` 一次性加载所有记录的模式阻碍了统一的流式架构。长期来看，需要将解析器重构为返回 `Iterator<Item = Result<Record>>` 的流式接口。
* 🟡 **`Value` 类型的过度包装 (中风险 - 关联 P6 deferred):** 依赖 `serde_json::Value` 限制了对内存布局的精细控制。

## 5. 剩余问题汇总与可操作建议

### 🔴 高优先级 (Immediate Action)

1. **统一流式处理架构 (性能与内存占用风险)**
   * **影响:** 目前处理大文件或使用 DSL 过滤时极易 OOM。
   * **建议:** 重构 `load_records`，使其返回一个实现了 `Iterator<Item = Result<Record>>` 的惰性迭代器。将 DSL 评估引擎改造为基于迭代器的 Pipeline，使得单纯的过滤、映射操作 (`where`, `select`) 在任何输入下都能保持 O(1) 内存。
2. **补充大文件与内存极限测试 (稳定性风险)**
   * **影响:** 缺乏对极限场景的保护网，容易在生产环境中出现未预期的崩溃。
   * **建议:** 编写针对大文件（通过生成器动态产生）的 E2E 测试，验证内存占用不会随文件大小线性增长。

### 🟡 中优先级 (Short-term Improvements)

1. **增强日期时间函数 (功能性风险)**
   * **影响:** 限制了 `qk` 在复杂日志分析场景下的表达能力。
   * **建议:** 在 DSL 和关键字层引入 `date_trunc` (截断到自然日、自然月) 以及时间属性提取函数 (如 `is_weekend`)。
2. **完善 E2E 流式测试 (稳定性风险)**
   * **影响:** `tail -f` 等流式场景在遇到异常数据或管道中断时行为未得到充分验证。
   * **建议:** 编写使用 PTY 终端或命名管道 (FIFO) 模拟真实流式数据写入的集成测试。

### 🟢 低优先级 (Long-term Bets & Deferred)

1. **优化 DSL 错误报告 (体验提升)**
   * **建议:** 使用类似 `miette` 或 `ariadne` 的库，为 DSL 解析错误提供带有源代码片段、下划线高亮和具体修复建议的错误报告。
2. **[DEFERRED] 字符串值驻留 (String Interning) (P6 架构优化)**
   * **影响:** 大量重复字符串值导致的内存膨胀。
   * **建议:** 当前由于 `serde_json::Value::String` 的限制无法实现。明确延后到下一个大版本，届时需引入自定义的 `Value` 枚举并结合字符串驻留池或内存映射 (`&[u8]`) 实现零拷贝解析。
