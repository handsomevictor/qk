# qk 全面审查与路线图

基于对代码库（文档、架构和源代码）的深入分析，以下是从高级主任工程师（Senior Staff Engineer）、开源维护者和具有产品思维的构建者的角度对 `qk` 的评估。

## 1. 项目理解

**核心价值:**
`qk` 是一个快速、原生的终端结构化查询工具，旨在统一零散的文本处理工具生态（`grep`、`awk`、`jq`、`yq`）。其核心主张是“一个二进制文件，一种语法，所有格式”，大幅降低了探索结构化日志和配置文件的认知负担。

**设计哲学:**
* **统一的中间表示 (Unified IR):** 将多种格式（NDJSON、CSV、YAML、TOML 等）规范化为扁平的 `Record` 中间表示（`IndexMap<String, Value>`）。
* **双层用户体验 (Two-Layer UX):** 提供一个快速的基于关键字的层（`where`, `select`），几乎不需要引号（覆盖 80% 的用例），并由正式的表达式 DSL（`.` 语法）支持复杂逻辑。
* **终端原生 (Terminal Native):** 内置语义化着色、Shell 安全的运算符（`gt`, `lt`）、自动格式检测以及透明的 gzip 解压。

## 2. 架构与代码审查

**优势:**
* 清晰、模块化的数据流水线（输入 → 检测器 → 解析器 → IR → 查询引擎 → 输出）。
* 很好地利用了现代 Rust Crate（`rayon` 用于并行处理，`memmap2` 用于零拷贝文件读取，`nom` 用于 DSL 解析器）。

**架构弱点与局限性:**
* **致命缺陷 - 全量物化 (Critical Flaw - Full Materialization):** `load_records()` 在评估之前将整个文件读取到 `Vec<Record>` 中（通过 `io::stdin().read_to_string`）。这会消耗 O(N) 的内存，使得在标准笔记本电脑上处理数 GB 的文件变得不可能，并完全破坏了 `tail -f` 流式处理。
* **性能陷阱 - 正则表达式重复编译 (Performance Trap - Regex Re-compilation):** 在 `src/query/fast/eval.rs` 和 `src/query/dsl/eval.rs` 中，`Regex::new()` 在评估期间被**每条记录**调用一次。对于一个 100 万行的文件，正则表达式会被编译 100 万次，严重破坏了性能。
* **内存膨胀 (Memory Bloat):** `Record` IR 使用 `IndexMap<String, Value>` 并且每条记录都存储原始字符串 (`raw: String`)。在数百万条记录中为相同的字段名（例如 "level", "latency"）分配字符串会导致严重的内存碎片和开销。

## 3. 功能与体验差距

**缺失的功能:**
* **时间序列聚合:** 原生缺乏按时间分桶日志的能力（例如 `count by 5m`），而这是日志分析的头号用例。
* **流式处理流水线:** 如上所述，目前的 `read_to_string` 实现从结构上就不支持真正的流式处理（`tail -f | qk`）。

**用户体验缺陷:**
* **错误报告:** 关键字层中的语法错误只返回简单的字符串，没有指向错误位置的直观提示。
* **逗号解析的边缘情况:** 使用逗号作为 `and` 的别名（`where level=error, service=api`）在语法上存在歧义，尤其是当用户在未加引号的字符串中搜索文字逗号时。

## 4. 文档审查

* **优点:** 对于一个早期项目来说，文档非常出色。`README.md`、`TUTORIAL.md` 和 `COMMANDS.md` 提供了清晰、可复制粘贴的示例。包含 `tutorial/` 测试数据目录对用户上手非常有帮助。
* **不足之处 (开源准备度):**
  * 没有 `CONTRIBUTING.md` 或 Issue 模板。
  * 没有用于自动化测试、代码检查和跨平台发布构建的 GitHub Actions CI/CD 工作流。
  * 用户目前必须使用 `cargo install --path .` — 没有预构建的二进制分发版或 Homebrew Tap。
  * 教程声称 `tail -f /path/to/app.log | qk where level=error` 有效，但当前代码在 `stdin` 上会无限阻塞，而不是逐行输出。

## 5. 产品潜力

* **演进方向:** `qk` 可以从一个简单的 CLI 演变成 SRE 和后端工程师的**默认本地日志浏览器**，作为一个轻量级的本地 Datadog/Splunk。
* **目标用户:** DevOps、SRE、后端开发人员和数据工程师（他们花费了太多时间链接 bash 管道）。
* **10倍体验功能:** 交互式 TUI 模式（`qk --ui`）。想象一下启动 `qk` 并获得像 `k9s` 或 `lnav` 一样的界面，你可以实时输入查询，立即看到结果，并在分页输出中滚动，而无需离开终端。

## 6. 竞争定位

* **vs. `jq`:** 对于 80% 的常见过滤/计数任务，`qk` 要简单 10 倍。它不需要复杂的引号或 `select()` 包装器。然而，`jq` 是一种图灵完备的数据转换语言；`qk` 则是严格只读和分析导向的。
* **vs. `grep` / `awk`:** `qk` 原生理解数据模式（嵌套 JSON、CSV 表头），而 `grep` 和 `awk` 将一切视为非结构化文本。然而，`grep` 和 `awk` 是真正的流式处理工具，可以在 10MB 的 RAM 上无限运行。
* **核心优势:** 符合直觉的语法、与格式无关的设计以及开箱即用的语义化着色。
* **当前劣势:** 缺乏流式处理能力，且处理大型数据集时内存开销高。

## 7. 可执行的路线图

### 快速制胜（未来 1-2 周）
1. **修复正则表达式性能 Bug:** 将 `Regex::new()` 移出评估循环，并在 `parser::parse` 阶段编译一次。将编译后的 `Regex` 存储在 AST 中。
2. **设置 CI/CD:** 添加 GitHub Actions 工作流以进行 `cargo test`、`clippy` 检查和自动化发布二进制文件生成。
3. **修复文档:** 从文档中删除 `tail -f` 示例，直到流式处理被实现，因为它目前会挂起。

### 中期改进（1-3 个月）
1. **流式执行引擎:** 重构 `load_records` 以返回 `impl Iterator<Item = Result<Record>>` 而不是 `Result<Vec<Record>>`。更新评估引擎以惰性处理记录。这将启用 `tail -f` 并大幅减少内存使用。
2. **字符串驻留:** 使用字符串驻留器将 `IndexMap` 中的 `String` 键替换为整数 ID，以节省重复字段名上的内存。
3. **分发:** 创建安装脚本 (`install.sh`) 和 Homebrew Tap，实现无摩擦的采用。

### 长期押注（3-6 个月）
1. **时间序列操作:** 添加原生日期时间解析和分桶（例如 `qk group_by time 5m`）。
2. **交互式 TUI:** 构建基于 `ratatui` 的交互式浏览器模式。
3. **数据富化/连接:** 添加使用本地 CSV 或 JSON 查找表来富化日志流的功能。
